# Arcane Docs site security and privacy review

## Review identity

- **Scope:** `apps/docs`, shared static-document, Markdown, browser-test, scoped-OPFS, and HTML-import cache mechanisms, deterministic public catalog builder, `dist/docs` package, and the GitHub Pages workflow.
- **Deployment profile:** public GitHub Pages project site at `https://thewizardnexus.github.io/ARCANE-OS/`; optional compatible Arcane host injection is a separate conditional path.
- **Data classification:** allowlisted repository documentation and screenshots are public. Search queries and assistant questions may reflect user interests and are treated as user data. Provider credentials are prohibited from the renderer and artifact.
- **Reviewer:** automated implementation review; independent security/privacy authority approval has not occurred.
- **Status:** suitable for local and public-site testing after the documented package and live-deployment checks; not a release-candidate approval.

## Assets, actors, and trust boundaries

| Asset or boundary | Actors and data | Required property | Implemented control |
|---|---|---|---|
| Repository to Pages artifact | Contributor, pull request, Actions runner; public Markdown, screenshots, runtime code | Undeclared, private, generated, case, credential, Git, cache, and machine-build files never publish | `public-content.json` is a positive document/screenshot inventory; the adapter copies into a fresh staging tree, generates hashes/sizes, rejects unsafe paths/links/case collisions, and verifies `dist/docs` before upload. |
| Catalog to renderer | Static JSON and Markdown; visitor browser | Only declared same-origin bounded bytes render after integrity verification | `StaticDocumentCatalog` validates closed records, resolves under one base URL, bounds fetch/time/UTF-8, and verifies exact byte size plus SHA-256 before hydration. |
| Markdown to DOM | Repository content and AI/reference text | No script, event handler, custom element, remote-image beacon, unsafe protocol, or DOM clobbering survives | `markdown-document` uses the bundled Marked path followed by a positive element/attribute/protocol allowlist and deterministic heading/link handling. |
| Shared Pages origin to executable component loader | Multiple `thewizardnexus.github.io` project sites; packaged component HTML and scripts | Origin storage cannot become executable publication authority; component fetches remain same-origin and redirect-free | `HTMLImport` never reads or writes executable component HTML in `localStorage` or OPFS. It fetches the resolved same-origin URL with credentials limited to same-origin and redirects rejected. The ready event keeps the literal `href` and adds `resolvedHref`. |
| Shared Pages origin to docs cache | Other same-origin sites, visitor, browser quota/eviction | Cache failure or mutation cannot become publication authority or clear unrelated data | `ScopedOPFSCache` binds the declared `docs` identity and exposes only exact-key JSON `get`, `set`, and `delete` inside `apps/docs/arcane-docs-public-catalog-v1`; it cannot list, export, restore, or clear a table/origin. Catalog version, hash, size, and text are reverified on read. Only public corpus data is stored. |
| Query to optional provider | Visitor question, strongest public excerpts, configured provider | No silent remote transmission, renderer-held key, tool execution, private checkout scan, or provider substitution | Ordinary Pages has no `Arcane.ai` bridge and stays fail-closed. A host must return a named, configured profile with explicit local/cloud status and expose `Arcane.ai.chat`. Cloud mode requires session consent. `ConfiguredAIChatSession` binds `expectedProvider`; context is bounded and labeled untrusted. No tool schema is supplied. |
| Git branch to Pages environment | Maintainer, Actions, GitHub deployment environment | Only reviewed main-branch output deploys with minimal token authority | Workflow triggers on `main` or manual dispatch, gives build no write permission, and grants `pages:write` plus `id-token:write` only to the dependent deploy job/environment. One-time repository Pages source configuration remains an administrator action. |

## Data flow and retention

1. The build reads only paths named in `apps/docs/public-content.json`.
2. It copies Markdown under `apps/docs/catalog/documents/`, copies the six selected PNGs under `apps/docs/screenshots/`, and creates a deterministic catalog plus release inventory.
3. The browser downloads the catalog. Search over titles, summaries, tags, paths, and headings remains in memory and sends no query to a server.
4. Document hydration fetches one same-origin published file, verifies it, and may cache the already-public text in the exact docs OPFS namespace. The app does not request persistent-storage treatment; browser eviction and site-data deletion apply.
5. Assistant conversation is memory-only. When a compatible host exposes a configured local provider, bounded public excerpts remain on that provider path. For a cloud provider, the visitor must check the displayed consent for the current page session before each session can transmit questions and excerpts. Provider-side retention, region, and account policy remain external deployment inputs.
6. The site never accepts a checkout upload, enumerates a visitor filesystem, stores a provider key, saves automatic memories, or exposes a shell, provisioning, package, or repository-write operation.

## Abuse-case review

| ID | Case | Prevention/detection/recovery | Residual risk |
|---|---|---|---|
| DOCS-SEC-01 | A broad deploy accidentally publishes case files, untracked data, repository metadata, build output, or secrets. | Positive inventory, fresh adapter staging, symlink/path checks, deterministic release verification, and deployment of only `dist/docs`. | A malicious change to the allowlist is a review problem; branch protection and CODEOWNERS are outside this change. Inspect the live artifact after first deployment. |
| DOCS-SEC-02 | Hostile Markdown runs script, injects a custom element, creates an event attribute, navigates to an unsafe scheme, or loads a tracking image. | Positive sanitizer and protocol rules; focused hostile-content tests; same-origin document routing. | Browser/parser changes and untested markup variants require ongoing adversarial tests. The component has no native capabilities on Pages. |
| DOCS-SEC-03 | A sibling Pages project writes a colliding or exact `HTMLImport` storage entry so Arcane Docs executes attacker-selected component HTML. | Accidental relative-key collision was reproduced first. Review then showed that absolute keys still allowed deliberate same-origin poisoning. The final repair removes executable component storage entirely, rejects cross-origin URLs and redirects, and has a hostile pre-populated-storage regression. | A compromise of Arcane Docs' served JavaScript or same-origin server content remains executable by definition. Package inventory, repository review, and Pages deployment integrity remain critical. |
| DOCS-SEC-04 | Another same-origin site tampers with or exhausts the docs OPFS cache. | Cache contains public data only and is non-authoritative; exact catalog hash/size/version verification turns mutation into a miss; failures fall back to verified network hydration. No whole-origin clear is exposed. | Availability can be degraded until browser site data is cleared. GitHub Pages cannot provide per-project origin isolation. |
| DOCS-SEC-05 | Oversized or redirected content exhausts the browser or escapes the project path. | Closed record limits, bounded record count/bytes/results/context, timeouts/abort, same-origin base containment, redirect rejection, and UTF-8/hash verification. | Full network-loss, quota-pressure, and slow-stream browser exercises remain live-test work. |
| DOCS-SEC-06 | Retrieved text prompt-injects the configured model or causes a tool action. | Context is explicitly labeled untrusted, added as transient user data, bounded to strongest public matches, and paired with a fixed app prompt. No tools or native operations are supplied. | Model output may still be wrong or manipulated. It is advisory and must cite source documents; provider/model adversarial evaluation remains required for a supported AI claim. |
| DOCS-SEC-07 | Public excerpts are silently sent to a cloud provider or a provider changes mid-session. | Complete profile/locality required, named disclosure shown, checkbox required for cloud, `expectedProvider` binding detects substitution, conversation is memory-only. | The injected host and provider retention policy require independent review. Pages mode intentionally provides no provider fallback. |
| DOCS-SEC-08 | Component, catalog, or test code uses visitor-authored JavaScript. | Browser test descriptors are fixed trusted functions; no editor or `eval`/`Function` path exists. Markdown scripts are removed. | Shared `HTMLImport` executes inline scripts from packaged same-origin component files by design, so the package inventory remains security-critical. |

## Findings and dispositions

| Finding | Severity | Evidence | Disposition |
|---|---|---|---|
| HTMLImport treated origin-wide storage as an executable component cache. Absolute URL keys removed accidental project collisions but could still be deliberately poisoned by any same-origin project and could retain stale code. | High | The first regression reproduced project A/B collision. Adversarial review demonstrated exact-key poisoning remained possible; the final `test/html-import-cache.test.mjs` preloads hostile HTML and verifies it is never read while same-origin network content loads. | Fixed by removing executable component storage, requiring same-origin URLs, and rejecting redirects. HTTP and service-worker caching remain the non-authoritative transport cache. |
| GitHub Pages does not provide custom response headers and `frame-ancestors` is not enforced from a meta CSP. | Low | Hosting-platform constraint; the app includes a restrictive meta CSP for supported directives and contains no authenticated/private UI. | Accepted for the public static site. Reassess if private data, authentication, or sensitive actions enter scope; use a host that can set headers for those claims. |
| GitHub Pages project sites for one owner share an origin, so an OPFS directory name is an application convention, not a security principal. | Informational | Browser origin model and project-site URL structure. | Only public, hash-verified cache data is stored. Never store credentials or private checkout content in this origin cache. |
| Optional AI depends on a future/injected trusted-host capability and provider policy not exercised by ordinary Pages. | Informational | `assistantReady()` requires profile, locality, configuration, and `Arcane.ai.chat`; GitHub Pages shows an unavailable state. | Fail-closed. Perform a separate host/capability/provider review before claiming supported AI on the public site. |

## Verification evidence and remaining gates

- The exact thirteen-file Pages/docs suite passed 86/86, including canonical app-data scoping, bounded async operations, component contracts, configured-provider binding, catalog publication, public packaging, Markdown sanitization, scoped OPFS, HTML import, route handling, and readiness failure paths.
- Natural-language catalog retrieval against all fourteen packaged documents selected `arcane-api` for `providerModels`, `restoreShell`, and `recentErrors`, and each bounded excerpt contained the requested API term.
- The packaged app loaded beneath `/dist/docs/apps/docs/index.html`, preserving project-site routes. Provisioning and developer guides rendered, local search returned the expected security result, all four live specimens loaded, all six screenshots decoded with nonzero width, and the in-browser fixed suite passed 6/6.
- Ordinary Pages mode displayed the explicit AI-unavailable boundary and exposed no provider request path. Current-run console inspection reported no warning or error associated with the packaged `/dist/docs/` origin.
- The built package must continue to pass `npm run app:check -- docs`; only `dist/docs` may be uploaded. The final artifact inventory and digest are recorded after the last clean package run.
- First GitHub Pages deployment still requires inspection of the actual public URL, artifact contents, platform CSP behavior, and repository Pages source configuration.
- No provider request, provider credential, native capability, privileged action, or private-data flow was exercised by this review.
- Independent security/privacy authority review and risk acceptance remain required for any release-candidate or supported injected-AI claim.
