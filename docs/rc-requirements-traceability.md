# ARCANE Release Candidate Requirements Traceability Matrix

> **Purpose:** This is the living ARCANE RC ledger for what is implemented, what is partial or absent, how each claim is proven, who owns the decision, and which ticket closes the gap. It is not a substitute for source, tests, or real-platform evidence.

## How to use this matrix

Update this document whenever a release requirement, supported claim, implementation status, evidence path, or RC disposition changes. The release owner freezes a candidate-specific copy or revision as part of the evidence required by `docs/rc-acceptance.md`.

Use only these implementation states:

| State | Meaning |
|---|---|
| Implemented | The required behavior exists on the supported boundary and has objective evidence. Candidate-specific evidence may still be required. |
| Partial | A meaningful foundation exists, but the complete RC behavior or proof is missing. |
| Simulated | The behavior is covered only through fixtures, mocks, source assertions, or transaction simulation. |
| Manually verified | The behavior was observed on the real supported platform, but repeatable automated proof is unavailable or inappropriate. |
| Absent | No implementation satisfies the requirement. Ideation, prompts, or adjacent behavior do not count. |
| Experimental | The implementation is deliberately outside the supported RC boundary. |
| Deferred | The requirement is not part of the first Microsoft NT RC claim and has an approved future ticket. |

Evidence must identify what it proves. A source path proves where logic exists. A focused test proves its asserted contract. A package check proves the generated package. A real Microsoft NT acceptance record proves installed and signed-in behavior. None silently substitutes for another.

## RC accountability roles

| Role | Accountable area |
|---|---|
| Product authority | Product boundary, supported journey, user control, and accepted exclusions. |
| Runtime engineering | Shared runtime, intent, capabilities, memory, policy integration, lifecycle, and diagnostics. |
| Platform engineering | Native hosts, Microsoft NT adapter, IPC, privilege, identity, session, installation, update, and recovery. |
| Application engineering | Thin application adapters, packaging declarations, and app-specific behavior. |
| AI engineering | Provider abstraction, model inventory, inference, resource governance, and model provenance. |
| Security and privacy authority | Threat model, authorization, privacy, adversarial evidence, and risk disposition. |
| Accessibility authority | Conformance baseline, assistive-technology evidence, and accessibility disposition. |
| Release owner | Candidate provenance, production signing, full evidence chain, operations, and go/no-go record. |

These roles are accountable placeholders. The RC acceptance record must bind them to named people before candidate approval.

## RC claim boundary from the current matrix

The current codebase can credibly claim substantial Microsoft NT runtime, packaging, privilege, identity, application, model-service, and installation foundations. It cannot yet claim the complete ideated ARCANE operating-system experience because typed intent planning/execution, enforceable TWiN Compass policy, general scoped memory, task-centric continuity, and their end-to-end evidence remain absent or partial.

The first implementation sequence should therefore prove one narrow governed vertical slice before expanding application breadth:

1. typed intent envelope;
2. reviewable plan and ambiguity resolution;
3. TWiN Compass decision and enforcement hooks;
4. governed capability execution;
5. scoped task/memory record;
6. unified confirmation, progress, action ledger, interruption, and recovery;
7. production-signed Microsoft NT reference-journey acceptance.

## Matrix review checklist

Before each RC review:

- [ ] Re-read every P0 row against the current source, tests, packages, and real-platform evidence.
- [ ] Replace role placeholders with named accountable people in the candidate acceptance record.
- [ ] Confirm every evidence path exists and proves the stated boundary.
- [ ] Separate automated, simulated, manual, development-signed, and production-signed evidence.
- [ ] Add newly discovered requirements and do not hide them inside findings or meeting notes.
- [ ] Update ticket links and status after implementation and clean-state verification.
- [ ] Record accepted P1 deferrals with impact, workaround, owner, deadline, and rollback trigger.
- [ ] Confirm P2 and experimental work is absent from supported RC claims.
- [ ] Freeze the reviewed matrix revision with the immutable candidate evidence.
