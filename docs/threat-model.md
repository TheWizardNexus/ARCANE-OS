# ARCANE Current-System Threat Model

## Purpose and status

This document is the consolidated threat-model foundation required by ARC-RC-100. It describes the implementation currently visible in this repository, identifies incomplete or unverified controls, and links residual work to the RC backlog. It is an analysis artifact, not a security approval, risk acceptance, penetration-test result, or release-candidate evidence record.

The implementation baseline inspected for this model is the repository's ARCANE 0.8.4 Microsoft NT machine bundle and shared Arcane runtime. Microsoft NT is the intended first RC System Platform. Linux support is experimental and deliberately fails closed for automatic privileged work. Product direction under `Ideation/` is not treated as implemented behavior.

Claims marked **implemented** are supported by source and focused repository tests. Claims marked **partial** require additional implementation or integration. Claims marked **candidate-unverified** require independent review or production-signed real-machine evidence for the exact candidate. No real-machine, production-signing, UAC, clean-install, assistive-technology, network-loss, or power-loss test was performed while preparing this document.

This model must be reviewed under [Security and Privacy Review SOP](security-privacy-review.md) and updated whenever a change affects a boundary below.

## Scope

### Included

- Packaged applications and their renderer processes.
- `window.Arcane`, the native host, framed Core IPC, and method dispatch.
- Microsoft NT identity, application capability, renderer-permission, privilege, package, storage, service, installation, and recovery boundaries.
- Local and remote AI-provider paths exposed by the shared runtime and machine bundle.
- Diagnostics, package inventories, publisher continuity, installation state, and recovery records.
- Current foundations relevant to future intent, policy, memory, update, and audit capabilities.

### Excluded or not yet established

- A public third-party extension ecosystem.
- A production TWiN Compass policy engine, intent planner/executor, or general scoped-memory service; these remain absent or partial in the [RC Requirements Traceability Matrix](rc-requirements-traceability.md). The shared v1 intent-envelope syntax contract is included, but no native ingestion or governed execution path is established.
- Linux privilege and session parity, mobile, vehicle, kiosk, and appliance deployments.
- Organization-specific legal, regulatory, data-classification, retention, incident-response, and risk-acceptance policy.
- Security properties of external AI, communications, identity, signing, timestamp, revocation, operating-system, and dependency providers beyond ARCANE's visible validation boundary.

## Protected assets and harmful outcomes

| Asset | Harm to prevent |
|---|---|
| Microsoft NT and Linux local users, stable SIDs or UIDs, credentials, profiles/homes, groups, login shells, and sessions | Impersonation, unauthorized account or shell changes, credential disclosure, lockout, service-account damage, or cross-user access. |
| Application identity and granted capabilities | Self-grant, confused-deputy execution, capability widening after elevation, or attribution to the wrong application. |
| Privileged request, confirmation, and broker session | Parameter substitution, replay, worker impersonation, privilege retention, or execution without exact user authorization. |
| User files, application storage, preferences, conversations, and future memory | Disclosure, cross-scope access, poisoning, deletion, provenance loss, or unapproved provider transmission. |
| Mail identities, SMTP credentials, recipients, report/crisis/chat bodies, and delivery state | Open-relay abuse, credential disclosure, delivery to an unintended person, duplicate or falsely confirmed notification, provider over-retention, or sensitive-content leakage. |
| Machine configuration, services, shell bindings, ACLs, and recovery journals | Persistent compromise, unusable shell, unsafe rollback, administrator escape, or irrecoverable partial state. |
| Application packages, machine release, signer identity, manifests, and installed baseline | Publisher substitution, downgrade, tampering, unsafe content inclusion, or activation of unverified bytes. |
| Core, shell, renderer, model runtime, provisioner, maintenance, and recovery availability | Denial of the supported desktop, inability to interrupt or recover, data loss, or unsafe fallback. |
| Diagnostics, audit evidence, and support artifacts | Secret or protected-content leakage, log injection, tampering, excessive retention, or false attribution. |
| User understanding and control | Authority spoofing, inaccessible or stale confirmation, misleading progress, confirmation fatigue, or inability to stop/recover. |

## Actors

- A normal ARCANE user acting correctly, accidentally, or after social engineering.
- An authorized administrator or support operator acting correctly, mistakenly, or maliciously.
- A compromised or malicious packaged application, renderer, document, message, web resource, or model response.
- An unprivileged local process attempting to impersonate Core, the native host, pipe guard, or elevated worker.
- A local administrator or physical attacker; some administrator-controlled policy is necessarily inside the trusted computing base, but misuse must remain attributable and recoverable.
- A compromised dependency, build worker, package source, signing/timestamp/revocation service, update medium, or publisher identity.
- A local or remote AI/communications provider that receives data, returns hostile content, is unavailable, or violates expected retention behavior.
- An authenticated web ingress or SMTP provider acting correctly, mistakenly, maliciously, or while unavailable; and an unauthenticated client attempting to turn the gateway into an arbitrary-recipient relay.
- A network attacker affecting provider, dependency, revocation, timestamp, download, or update traffic.
- A failure source: crash, hang, resource exhaustion, network loss, power loss, disk corruption, clock change, or partial durable write.

## Current strengths

- Authorization is enforced in Core rather than treated as a hidden UI control, and elevation does not intentionally widen the application's declared authority.
- The Microsoft NT privilege path binds kernel-observed process identity to an exact cryptographic request/session/application context and has focused adversarial tests.
- Application and release packaging use positive inventories and explicit integrity checks rather than broad directory publication.
- Native application storage is scoped, validated, atomic, quota-bounded, and split into read/write grants.
- Recovery code contains concrete ownership, ACL, reparse-point, changed-state, and shell-binding defenses.
- Linux automatic privileged work is disabled instead of silently using a weaker peer boundary.

## Priority residual risks

1. **Governed-action boundary:** typed intent, reviewable plans, exact confirmation, TWiN Compass decisions, and enforcement hooks are not yet a production security boundary (ARC-RC-020–023, ARC-RC-030–032).
2. **Memory and data release:** general scoped memory, promotion, provenance, user control, deterministic de-identification, and remote-provider release controls are incomplete (ARC-RC-040–043, ARC-RC-062).
3. **Candidate proof:** strong source and test foundations still require independent review and a production-signed disposable Microsoft NT machine exercise for identity, UAC, package, install, shell, failure, recovery, and rollback (ARC-RC-090, ARC-RC-092, ARC-RC-102).
4. **Operational visibility:** the correlated action ledger and privacy-safe support bundle are incomplete (ARC-RC-081–082).
5. **Update and maintenance:** authenticated maintenance, health-gated update activation, known-good rollback, offline parity, and power-loss behavior are incomplete (ARC-RC-071, ARC-RC-083, ARC-RC-091).
6. **Accessible security:** security confirmations, errors, interruption, and recovery have not completed the required assistive-technology journey evidence (ARC-RC-110–111).

### Native system-platform presentation and permission messaging

- **Assets and people:** users' understanding of which operating-system family is active, whether protected operations can be authorized, whether a release is publisher-verified, and whether an installation action is actually available. Misleading elevation copy can cause unsafe workarounds or false confidence.
- **Trust boundary:** native `platform.status` and permission evidence cross the renderer bridge into the shared `SystemPlatformPresentation` helper and the Provisioner. The helper writes a display name, `data-arcane-kernel`, and an `arcane-kernel-*` class to the document root. These DOM values are renderer-controlled presentation state, not host evidence.
- **Controls and evidence:** technical `windows`/`win32` identifiers remain stable at the native/API boundary; the shared helper maps them to the Microsoft NT display name. Both cached machine status and live platform status use one Provisioner rendering path. Explicit `canElevate: false` produces a no-automatic-elevation state; only native permission evidence controls privileged dispatch. The helper is included in exact release inventories, and automated tests cover marker replacement, stale-class removal, Microsoft NT UAC copy, Linux fail-closed copy, elevated/root states, and accessible live announcements.
- **Abuse cases and residual risk:** script injection or a compromised renderer could alter classes or visible text, while stale cached status could briefly misdescribe the host. Native capability checks, release verification, privileged-worker authentication, and installation postconditions must therefore ignore all DOM markers. Linux publisher verification and kernel-enforced peer authentication remain absent, so Linux protected actions stay unavailable even if renderer state is forged. This change is not production signing evidence, security approval, or release-candidate acceptance.

### Linux unsigned-local protected installation staging

- **Assets and people:** the administrator-authorized release bytes, `/opt` installation parent, executable entry points, installed ownership and modes, release source files, rollback target, and unrelated local files that must not be reached through a link or cleanup mistake. A user-owned or mode-`0777` release must not make the privileged stage writable, transfer source ownership or special metadata, or cause Arcane to mutate an outside inode.
- **Trust boundary:** an explicitly allowed unsigned-local Linux release is readable by an already-root Core, but its filesystem metadata remains untrusted. Only manifest-listed bytes cross into a new stage under the protected installation parent. The stage identity is captured before population and remains mode `0700` until its complete file and directory topology is canonicalized.
- **Controls and evidence:** Core creates the stage with a private mode before any copy and orders prepare, identity capture, native materialization, generated launchers, finalization, byte-integrity verification, protected-tree verification, rename, and an exact post-rename identity check before treating activation as successful. The Linux adapter validates the canonical stage name and root-owned/non-writable real parent. Before staging, it retains no-follow directory descriptors, resolves descendants through `/proc/self/fd`, reads the manifest and payload through bounded descriptors, and rechecks the path-to-descriptor directory and file identities around inventory and content reads. It then byte-copies only inventory entries through `O_NOFOLLOW` source descriptors into `O_CREAT|O_EXCL|O_NOFOLLOW` destination descriptors; bounds copied size; checks SHA-256 while copying; rechecks every source-parent identity; creates no file from source metadata; and requires fresh single-link root-owned files. Device and inode values used as security identities come from BigInt stats and are compared as canonical decimal strings. Launchers use the same exclusive writer. Before exposure it rejects extra files/directories, links, special files, cross-device entries, changed identities, or noncanonical modes; changes directories from private `0700` to `0755` through identity-checked handles; and exposes the root last. Exactly six program paths are `0755`; all other files are `0644`. Failure cleanup accepts only the captured stage identity (or its exact failed-activation rename). Focused portable and root-WSL tests reproduce the original user-owned `0777` source, final-file and intermediate-directory replacements, identities above JavaScript's safe-integer range, and hardlink injection.
- **Abuse cases and residual risk:** the unsigned-local consent is not publisher authentication, and a malicious root, compromised kernel/filesystem or procfs, storage fault, or power loss can defeat application-level controls. Descriptor-rooted traversal and pre/post identity checks fail closed for covered replacement races, but this is not Linux `openat2(RESOLVE_BENEATH|RESOLVE_NO_SYMLINKS)` proof on every supported kernel. ACL, xattr, capability, immutable-flag, crash-recovery, disk-full, concurrent-root, update/rollback, and clean-machine distro behavior still require disposable-host and production-signed candidate testing. This implementation is not production signing, security approval, RC acceptance, or residual-risk acceptance.

### Linux graphical-session and default-target installation

- **Assets and people:** the machine's boot target, display-manager login path, administrator recovery choice, Arcane session registration, and the authenticated user's graphical session. A wrong or attacker-controlled target/session could prevent normal login, launch untrusted code, run the GUI with excessive privilege, or strand a headless/WSL installation.
- **Trust boundary:** a publisher-verified Linux release and separately authorized root installation process may cross from Arcane's staged payload into `/usr/share/xsessions` and systemd's protected default-target state. The registered session later crosses back to the unprivileged authenticated user. WSLg is a separate per-application boundary and is not treated as a display-manager boot environment.
- **Controls and evidence:** the release manifest carries an exact first-install Linux policy. The installed tree contains a fixed root-owned session wrapper; the adapter refuses symbolic, non-root-owned, writable, changed, or missing wrapper/session files. It detects WSL before any systemd call, requires systemd as PID 1, verifies loaded `graphical.target`, loaded and enabled `display-manager.service`, and uses one absolute `systemctl` executable with constant argument arrays and no shell or nested `sudo`. The prior target is validated and persisted, the exact postcondition is re-read, command/postcondition and later install failures restore the prior target, and update/repair paths preserve later administrator choices. Only an X11 session is registered; Arcane does not misrepresent its client window as a Wayland compositor. Focused functional/source tests cover these branches and rollback order.

Linux automatic elevation remains disabled; this mutation must stay unreachable from the normal GUI until publisher verification and an already-root or kernel-peer-verified privilege boundary exist. No production-support, security-approval, or RC claim follows from this implementation.

### Linux local-user provisioning

- **Assets and people:** standard local accounts, exact UIDs, password hashes and temporary credentials, group membership, home directories, POSIX login shells, protected recovery state, console/SSH availability, and the administrator's ability to restore the prior shell. The current/elevating account, root, system/service identities, and administrative accounts must never be converted by this workflow.
- **Trust boundary:** an integrity-verified Arcane release running from a separately authorized root session crosses into the distribution's root-owned account databases and tools. Account activation is a separate privileged request after credential delivery. WSLg shares the Linux account database but does not become a display-manager session or gain automatic elevation.
- **Controls and evidence:** Core keeps the existing provisioner-only `users.manage`, exclusive-mutation, staged-account journal, separate activation, password-apply, and shell-restore contracts. The Linux adapter constrains username syntax, UID range, unique UID, standard/non-administrative groups, normal prior shell, local absolute home and exact home owner; protects the current/elevation identities and known system names; revalidates root-owned single-link account/state files, the shell registry, and the installed shell wrapper; permits hardlinked executable inodes only when their candidate and canonical directory chains are root-owned and non-writable by group/others and the canonical inode is root-owned, executable, and non-writable by group/others; and skips an ineligible candidate only in favor of a separately validated canonical candidate. The integrity-backed install stage establishes exact metadata before activation, Core binds Linux activation to the captured stage identity, and post-activation integration no longer runs recursive `chown` or `chmod`. Exact argument arrays, a sanitized subprocess environment, protected standard-input credential delivery, locked staging, exact UID binding, and verified shell/lock postconditions remain in place. Focused native-adapter, executable-selection, Core transaction, and Provisioner accessibility-contract tests cover these properties without storing a credential or mutating a real account.

Source and simulated tests do not prove real `/etc/shadow` locking, distro-specific `useradd`/`usermod` behavior, power-loss atomicity, remote graphical-session semantics, WSLg display-variable behavior, display-manager account selection, login/password-change flow, or rollback on a disposable host. Automatic Linux elevation and production publisher signing remain unavailable, and accessibility evidence for the complete create/credential/activate/login/restore journey is absent. This experimental unsigned-local capability is not production support, security approval, accessibility conformance, RC acceptance, or risk acceptance.

### Android HOME-role boundary

### PreCrisis initial speech-mute preference
