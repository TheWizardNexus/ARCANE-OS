This document explains Ollama's content-addressable model storage system, including the manifest/layer architecture, blob storage, layer types, and registry operations. It details how models are composed of layered blobs, the organization of these blobs in a registry filesystem, the handling of different layer media types, and the logic for remote push/pull operations.

For in-depth details about writing Modelfiles which define model composition, see Modelfiles. For file format conversion and quantization topics, see Model Conversion and Import and Quantization.

Overview

Ollama’s model storage is based on a content-addressable storage system inspired by Docker’s layer distribution mechanism. The core concepts are:

  • Manifest: A JSON descriptor referencing an ordered set of layers. It describes the model structure and metadata. server/internal/client/ollama/registry.go47-53
  • Layer: Individual immutable content blobs identified by a SHA256 digest. These can be model weights, prompt templates, system messages, or model-specific parameters. server/internal/client/ollama/registry.go101-106
  • Blob Storage: The physical storage location for layers, organized by content digest. server/internal/client/ollama/registry.go76-84
  • Registry: A remote or local service that stores manifests and associated layers, enabling push/pull operations. server/internal/client/ollama/registry.go1-4

This design enables efficient storage through deduplication of content shared across models, atomic updates via manifest replacement, and straightforward distribution.

Sources: server/internal/client/ollama/registry.go1-106

Storage Architecture

Directory Structure

Models and their components are stored in a layered structure under the user's Ollama data directory (default ~/.ollama). The layout is:

~/.ollama/
├── manifests/
│   └── registry.ollama.ai/
│       └── library/
│           └── llama3/
│               └── latest          # Manifest JSON file describing a model
└── blobs/
    ├── sha256-abc123...           # Model weights blob (GGUF file)
    ├── sha256-def456...           # Config blob (JSON)
    ├── sha256-789xyz...           # Template blob (text)
    └── sha256-...                 # Other layers
  • manifests/ stores manifest JSON files organized by registry domain, namespace, model name, and tag.
  • blobs/ stores the immutable content blobs named by their SHA256 digest prefixes. server/internal/client/ollama/registry.go76-84

Sources: server/internal/client/ollama/registry.go76-92

Content Addressing

Each layer blob is uniquely identified by its SHA256 hash (digest). The digest is computed over the file contents, ensuring:

  • Integrity: Content validation against the expected digest.
  • Immutability: Once written, layers are never modified.
  • Deduplication: Identical content is stored once.

The blob.DiskCache manages these files, and blob.Digest represents the SHA256 identifier. server/internal/client/ollama/registry.go39-40

Sources: server/internal/client/ollama/registry.go11-16

Manifest Structure

A manifest file is a JSON document that describes a model by referencing a config layer and an ordered list of other layers. The manifest schema is modeled in Go as ollama.Manifest. server/internal/client/ollama/registry_test.go47-58

Manifest to Layer References

The manifest fields include:

  • Config: A reference to a JSON config blob describing model-specific metadata such as OS, architecture, capabilities, and root filesystem type. server/internal/client/ollama/registry_test.go54-57
  • Layers: An ordered array of Layer objects, each detailing a content blob with digest, size, and media type. server/internal/client/ollama/registry_test.go131-136

Sources: server/internal/client/ollama/registry_test.go47-58

Automated Template Detection

When a model is created, Ollama attempts to detect the appropriate chat template by inspecting the GGUF KV pairs (specifically tokenizer.chat_template). If found, it automatically creates a template layer. server/model.go93-122

Sources: server/model.go23-122

Registry Operations

Ollama supports pushing and pulling models from remote registries following the manifest and layer model. The Registry struct in server/internal/client/ollama/registry.go provides the client implementation. server/internal/client/ollama/registry.go189-234

Pulling Models

The pull process involves multi-part parallel downloads for large blobs.

  1. Request Manifest: Fetch the JSON manifest for the requested tag.
  2. Download Blobs: For each layer, the blobDownload manager initiates a transfer. server/download.go41-55
  3. Chunking: Large blobs are split into up to 16 parts (default 100MB-1000MB each) and downloaded concurrently using errgroup. server/download.go99-103 server/download.go276-283
  4. Sparse Files: On supported systems, Ollama uses sparse files (setSparse) to allocate space for large model weights before the download completes. server/download.go225-227

Pushing Models

  1. Prepare: The server checks if the blob already exists on the remote using a HEAD request or a cross-repository mount. server/upload.go55-92
  2. Parallel Upload: Similar to downloads, blobs are uploaded in 16 parallel parts using HTTP PATCH with Content-Range headers. server/upload.go49-53 server/upload.go218-226
  3. Commit: After all parts are uploaded, a final HTTP PUT request with an etag (MD5 of parts) and the digest commits the blob. server/upload.go182-193

Authentication

Registry operations use a challenge-response authentication flow.

  1. The client receives a WWW-Authenticate header containing a realm, service, and scope. server/auth.go22-26
  2. The client requests a token from the realm (usually https://ollama.ai/token) by signing the request with the user's Ed25519 key. server/auth.go53-73
  3. The resulting JWT token is used in the Authorization: Bearer <token> header for subsequent blob transfers. server/auth.go94-99

Sources: server/download.go41-283 server/upload.go49-226 server/auth.go22-99
