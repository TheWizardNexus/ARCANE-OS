# Arcane OS 0.8.4 Validation Record

The canonical portable bundle gate is:

```bash
npm run check
```

It generates the Core and application payloads, runs source verification and the full smoke suite, tests the target packager, builds every registered target with `--platform=portable`, and verifies each exact target inventory. From the repository root, `npm run check` first runs every shared Arcane test and then invokes this bundle gate. The repository pre-push hook runs that portable command followed by `npm run check:windows`, which compiles and verifies the Microsoft NT release, the real pipe guard, native app targets, and dispatch contracts. The `Arcane checks` GitHub Actions workflow repeats the portable and native Microsoft NT gates for pushes and pull requests.

The automated 0.8.4 coverage includes:

- Building an exact Android debug-local-test distribution containing one HOME-eligible Launcher APK plus 17 independently installable application APKs, with a manifest binding package names, byte sizes, SHA-256 hashes, and per-app network policy.
- Running API 35 emulator instrumentation for Launcher HOME eligibility, hardened WebView bootstrap, the 17-app installed catalog, Browser navigator configuration and remote home-frame load, and Terminal execution of a real command through `/system/bin/sh` under the Terminal app UID.
- Generating `runtime/arcane-core.cjs` and the built provisioner and shell payloads from their shared sources.
- Parsing the generated Core, external `arcane-api.js`, and every inline SPA script.
- Verifying the WebView2 `Send(requestJson)` transport, preservation of native bridge diagnostics, the Microsoft NT host source preflight, and compiled COM `IDispatch` names when a Microsoft NT native build runs.
- Exercising framed `arcane/1` RPC and the simulated non-elevated-to-privileged flow, including automatic continuation of the original request.
- Rejecting a disclosed-token first client that claims a different process id and rejecting an invalid broker signature. The successful protocol path uses the source implementation's signed session/request/key binding and X25519/HKDF-derived AES-256-GCM request, event, and response frames.
- Verifying the Microsoft NT privilege boundary source/build contract: `ArcanePipeGuard.exe` must be compiled, inventoried, and used before the Node broker accepts a worker; production Core/guard executables must share one valid Authenticode signer, while the explicit `--allow-unsigned-local-release` test mode still requires an unsigned sibling pair bound by the exact schema-2 release manifest. Linux automatic elevation must fail closed while no `SO_PEERCRED` guard exists. Each Microsoft NT native build also runs the compiled guard against an attacker that claims the legitimate worker PID, proves `GetNamedPipeClientProcessId` rejects it, and confirms that only the kernel-matched client bytes are relayed.
- Verifying the purpose-specific, bounded WinTrust policy and provider-state lifecycle: strict-online, no-network baseline, and cache-only revocation use distinct provider flags; provider/subject-form errors are invalid rather than unsigned; true unsigned mode requires an empty PE certificate table; the selected signer and timestamp chain have error-free provider chains and certificate-chain policy; and provider state is always closed. Each trust call runs in an isolated native worker with a minimal environment, bounded output, and monotonic deadline; the compiled dispatch test forces a timeout and proves the worker is killed and reaped. Canonical degradation requires a recent exact protected attestation, standalone targets and the Provisioner stay strict online, wall and monotonic clocks enforce expiry, and background retries keep only genuinely unavailable sessions degraded.
- Confirming per-application capability grants and denials, including denial of provisioner-only methods to the shell, denial of app storage to the provisioner, and repeated authorization enforcement inside the elevated worker.
- Exercising app-scoped storage set/get/list/delete behavior, safe key validation, and application isolation in simulation.
- Exercising Microsoft NT and Linux account provisioning: new accounts remain disabled while credentials are returned, `users.activate` is a separate request, existing accounts retain their passwords, and platform recovery binds to the exact SID or UID rather than a username.
- Injecting failures after account creation, profile creation, shell assignment, and durable state writes; simulating crashes before SID return, while activation is pending, during activation, and after Microsoft NT enables the account; and verifying exact-SID rollback, fail-closed recovery, retryable activation, and shell restoration.
- Preserving a journal-backed restore affordance when a signed-out Microsoft NT profile can only be reported as `recorded-only`; the restore remains privileged and reloads the profile to reverify both exact shell bindings before making any change.
- Verifying the two-request existing-account password reset: `users.resetPassword` returns a generated credential without mutating the operating-system password, and only a later privileged `users.applyPassword` applies that saved value. The RPC and command-hardening tests also verify the phase boundary and that Microsoft NT passwords travel on protected standard input rather than in a command line or generated script.
- Verifying release schema 2 fixtures for Microsoft NT and Linux with exact file inventories, byte sizes, and SHA-256 hashes; changed assets, extra files, and unsafe paths fail closed.
- Building Linux through an isolated atomic `dist/linux/` publication so Microsoft NT, Android, portable-app, and temporary outputs cannot contaminate its exact release inventory; the completed build re-verifies that published directory before replacing the prior Linux output.
- Requiring the exact `build:distribution:linux:unsigned-local-test` publication flavor, its matching `verify:distribution:linux:unsigned-local-test` gate, and explicit `--allow-unsigned-local-release` native-host consent. Linux security tests prove ambient release claims are stripped, a bare switch cannot invent the sanitized host claim, installed launchers preserve only that same explicit mode, exact staged and installed inventories are reverified, no publisher attestation is created, and signed or attested installations cannot be downgraded. Focused regressions replace a same-byte final file and an intermediate directory during descriptor verification, compare filesystem identities above JavaScript's safe-integer range without rounding, and require fail-closed readiness and cleanup. Root-only WSL regressions additionally start from a non-root-owned mode-`0777` release, require a private identity-bound stage and fresh manifest-directed byte copies, enforce exact root ownership and `0755`/`0644` modes, and reject a hardlink injection before changing its outside inode. Core source-order checks require the same identity again after activation.
- Verifying that `SystemPlatformPresentation.js` is included in every native application inventory, maps technical `windows`/`win32` status to the Microsoft NT display name, and applies exactly one presentation-only `arcane-kernel-nt` or `arcane-kernel-linux` root marker. Provisioner UI tests cover standard/elevated Microsoft NT and standard/root Linux states, require accessible live permission messaging, and prove that Linux never promises UAC or on-demand automatic elevation when `canElevate` is false.
- Verifying the Linux graphical-session policy with functional adapter tests: WSLg performs no mutation; non-root, non-systemd, missing-target, missing-display-manager, and unverified-session cases fail before mutation; a supported fresh install uses exact absolute `systemctl` argument arrays, verifies `graphical.target`, records the prior target, and restores it after a failed postcondition or later installation rollback. Source-order tests prove updates and repairs do not reapply the first-install policy, no nested `sudo` exists, and only an X11 display-manager session is registered.
- Verifying the complete installed-tree inventory independently, including same-size SHA-256 tampering, unexpected files, obsolete integrity metadata, and traversal entries.
- Creating the publisher attestation through a self-verifying strict native probe inside the staged install transaction, validating exact probe/stage signer and machine/app bindings, rejecting stale or changed signer/binding/time/trust-source evidence, enforcing ACL-validated 64-bit HKLM or installed signer continuity and explicit predecessor-bound rotation, binding top-level `securityMode`, rejecting signed-to-unsigned downgrade, and proving signed installed enumeration performs no uncontrolled PowerShell Authenticode call before the Core is ready. Legacy migration tests accept only an older pre-integrity product under secure inherited Program Files ACLs, enumerate every executable independently, and reject equal/newer/mismatched identity plus mixed, invalid, untimestamped, or divergent publisher evidence.
- Verifying runtime navigation isolation, generated default-deny Content Security Policy, exact inline-script hashes, and Permissions Policy. The built-in shell retains microphone-only access for its trusted origin while the provisioner denies microphone access.
- Verifying trusted absolute command resolution, a sanitized subprocess environment, and protected standard-input handling for Microsoft NT temporary passwords.
- Injecting failures before and after installation activation, cleaning only the cryptographically named stage whose filesystem identity still matches the directory Arcane created, and preserving replacement or unrelated recovery trees when identity cannot be proven.
- Building isolated BOSS and PreCrisis targets, checking URL relocation and every local dependency, injecting per-target CSP/Permissions Policy, recording traversal-safe nested navigation allowlists, confirming deterministic exact inventories with no symbolic links, preserving other verified projections during targeted rebuilds, and rejecting unsafe paths, privileged capabilities, and unknown apps.
- Enforcing the BOSS publication boundary: the 0.8.4 package contains an `empty-unpublished` zero-record catalog and no Markdown corpus; records cannot cross the package boundary without separate publication authorization.
- Running the development-only HTTP fallback and an end-to-end `system.ping` request while keeping stdout protocol-only and diagnostics on stderr.

The requirement model is platform- and scope-aware: packaged applications treat Node.js as nonblocking, while Arcane-user provisioning requires a healthy machine-wide Ollama runtime. Microsoft NT preserves that requirement during base installation, distinguishes a user-only Ollama copy from the protected global executable and `ArcaneOllama` service, validates the service command, automatic start mode, and running state, and exposes the verified official-archive install/repair action. Linux and WSL report missing or unhealthy Ollama as nonblocking for base Arcane OS installation while retaining the requirement before local-AI use; Linux installation remains manual. Portable checks use fixtures and perform no runtime downloads or third-party installation.

The portable checks intentionally do not change a real user, assign a real login shell, lock or log out the active session, approve a real elevation prompt, or launch a native target executable.

## Controlled Microsoft NT acceptance sequence

The local unsigned build may be used only as a deliberately approved acceptance artifact. The operator must explicitly approve launching `dist\nt\bin\ArcaneProvisioner.exe --allow-unsigned-local-release`; building or inspecting the release is not launch approval. Never bypass Microsoft SmartScreen. Stop if the displayed version, release mode, installation path, protected username, or requested elevation differs from the expected test plan.

For the `arcane-test` acceptance account:

1. Before launch, record the existing `codex` and `arcane` account SIDs, enabled states, group memberships, and password-change timestamps; confirm that `arcane-test` does not exist. Record the installed Arcane version and release mode without changing either installation or account state.
2. Launch only the verified 0.8.4 Provisioner from `dist\nt\bin`, confirm its explicit unsigned-local warning, and install or update Arcane through the normal UI. The operator approves the real UAC prompt; automation must not approve it.
3. Add only `arcane-test`. Confirm that the Provisioner identifies the current administrator account as protected and does not offer to replace its shell.
4. Stop when the new account credential is displayed. The account must still be disabled and marked `activation-pending`. The operator saves the temporary password privately; do not copy it into diagnostics, terminal output, the clipboard automation stream, source files, or the test report.
5. Verify that `arcane-test` has a new stable SID, belongs only to the standard local Users group plus platform-default implicit principals, has a profile, and has both Arcane shell bindings recorded. Recheck that the `codex` and unrelated `arcane` account snapshots are unchanged.
6. Only after the operator confirms the credential is saved, select **Activate staged users** and approve its separate UAC prompt. Verify that the same SID is now enabled, remains non-administrative, and the protected recovery journal is complete.
7. The operator switches to `arcane-test`, enters and privately changes the temporary password at first sign-in, and confirms that Arcane Shell becomes the fullscreen desktop without another UAC prompt. Do not automate the password entry or password-change UI.
8. From that session, verify the displayed 0.8.4 release identity and unsigned-local warning; launch BOSS and PreCrisis from the installed catalog; exercise their allowed storage, network-status, diagnostics, identity, requirements, and system-read APIs; confirm privileged Provisioner-only APIs remain denied.
9. Exercise lock/unlock, logout/login, renderer or Core failure recovery, and a deliberately hung initialization on the disposable account. Explorer must be restored before the unusable Shell is terminated.
10. Return to the administrator account and verify the installation, account SID/state/groups, both shell bindings, journal, backup, and stage directories. Restoring the shell, resetting a password, deleting the test account, or deleting preserved recovery directories requires a separate explicit operator decision.

The acceptance result is incomplete until every expected state is observed from the real `arcane-test` session. A simulated transaction, source assertion, compiled dispatch test, or successful unsigned-local installation alone is not evidence of real-login acceptance. Production signing remains separately incomplete until a genuinely signed and RFC 3161-timestamped release passes the signed-machine cases above.
